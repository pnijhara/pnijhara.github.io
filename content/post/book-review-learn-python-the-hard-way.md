**Learn Python the Hard Way - Zed Shaw** was the first book anyone gifted to me. The book was gifted to me by my Computer Science teacher when I left school. The book wasn't much of a help to me at that time as I already learnt programming in Python in my school days. However, when I entered college, Rahul and Areeb Suggested I read that book. I read the book during my summer holidays back when I was an intern at Ajira Tech.

So, here is my review of the book. 

> Note: Before starting, I wanted to be clear that no book in the world is perfect. There are some people for whom this book is very close. I am just stating what I liked in this book and what else the author could do to make this book better. This book may also be a spoiler for some people.

<!--more-->

## About the author
Zed A. Shaw is an experienced software developer known for his Ruby applications (most notably, the Mongrel webserver), tech articles, and educational projects (e.g. a series of Learn Code The Hard Way courses). However, he owes his popularity not only to his books and projects; Zed is also an outspoken critic of certain technologies and aspects of the programming culture. He has written some popular blogs/essays outlining the problems that Python3 3 have. According to him "There is a high probability that Python 3 is such a failure it will kill Python".


## Book's Structure
According to the author, as written in the preface of this book, this book has some techniques that are beneficial for beginners who know nothing and need to acquire basic skills before they can understand the complex code structures.

Unlike other books which are broken into separate chapters focussing on data types, loops, functions, OOPs etc. This book is broken into several small exercises. 

- Exercise 0 contains the setup part for installing and using Python in MacOS, LINUX and Windows OS.
- Exercise 1-14 are dedicated to the basics of Python, strings, variables and printing.
- Exercise 15-17 contains details about functions

Now, I am going into much details about the chapters/exercises. So, basically till exercise 39, Zed tried to write as much as he could about basic Python till dictionaries and lists. In the second last section of the book, *which I recommend as a read section* Zed covered Object-Oriented Programming. Zed has written this section in a manner that no author can write.

And lastly, which I think very few books cover when it comes in terms of basic Python programming is brief introduction to web-development with Flask.

## Why Learn Python "the hard way"?
The main objective of this book was to get anyone started in programming. The title says it is the hard way to learn to write code, but it is not. It is only the “hard” way because it uses a technique called instruction. Instruction is where I tell you to do a sequence of controlled exercises designed to build skill through repetition. This technique works very well with beginners who know nothing and need to acquire basic skills before they can understand more complex topics. It’s used in everything from martial arts to music to even basic math and reading skills.

So basically, *The hard way is the "learn it yourself approach".*

## What I like in this book
The best section that I found in this book is "Common Student Questions". Realizing that many readers would encounter various bugs, mistype, and make other mistakes, Zed includes a ”Common Student Questions” section in every chapter. As the name suggests, it explains how to solve the most typical problems that readers might encounter — this is, essentially, Stack Overflow baked directly into the book!

Other than the question/answer section this book also provides a mindset of how to think like a software developer which is very uncommon in these type of books. The exercises are also great and approachable. Unlike other approaches in which a lot of instructors use abstract values etc, this book uses examples you can understand to begin to connect the dots.

## What I didn't like in this book
Way too much time is spent at the beginning of the book teaches you how to print things. What Shaw covers in the first 10 exercises could be covered in 2-3 even maintaining the very simplistic nature of the text. He then dives into manipulating files, a subject that doesn't need to be introduced until much later and only needs one lesson for the beginner. By contrast, tuples which is a very important concept in Python only get one paragraph very late in the book. It's the same with Objects/Classes and OOP in general. All of these concepts are forced into 3-4 lessons late in the text and Shaw seems way more concerned about the reader using the correct terminology than he is about actually explaining practical uses for the concepts.

The beginning few chapters weren't bad/hard but later chapters are harder to understand what I was doing because it doesn't explain why something is done. The extra questions at the end of each chapter aren't always helpful.

Lastly, this book isn't intended to motivate you. If you feel low you are allowed to step back. But if you have self-motivation then this book will force you to use search engines to search more and more.

## Conclusions
I think this book is positively influenced by Zed's background as an artist and a critic. As you can probably tell, Zed’s criticism of Python 3 was reflected in Learn Python the Hard Way — and this lead to another chain of criticism (now of Zed) from Python 3 supporters. In the book, it was seen Zed’s radical approach was justified.

Overall, I recommend this book as an introduction to learning to code and Python specifically for new or early programmers. Evaluating the book on what I believe to be Zed's own goals, the main flaws are in the sudden leaps from meticulously explained code examples to quizzes that expect you to do independent research and implementation. There wasn't much in the way of "intermediate" tasks in the code drills. There's some half-hearted name-dropping, but there's not much guidance for readers who want to learn more than what's covered in the book. To draw a contrast, we name things for what they are in the Haskell Book and footnote many references in addition to recommending further reading at the end of each chapter.
